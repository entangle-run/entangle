# OpenCode Session Continuity Slice

## Current repo truth

Entangle already keeps OpenCode as the default coding engine profile and
captures `engineSessionId` from `opencode run --format=json` output. Before
this slice, the captured id was only projected for observability; later turns
for the same Entangle session launched fresh OpenCode sessions.

The local OpenCode reference under
`/Users/vincenzo/Documents/GitHub/VincenzoImp/entangle/resources/opencode`
shows that `packages/opencode/src/cli/cmd/run.ts` supports `--session` for
continuing an existing session, and that attached or local `opencode run`
still emits JSON events containing the OpenCode `sessionID`.

## Target model

Each Entangle node should get better coding-agent continuity without letting
OpenCode become Entangle's orchestration engine:

- Entangle still owns graph sessions, messages, assignments, policy, artifacts,
  wiki state, and User Node communication;
- the OpenCode adapter may maintain engine-local continuity per Entangle
  session id;
- the adapter should pass `--session <opencode-session-id>` on later turns
  when it has a previous mapping;
- the mapping must remain node-local engine state under the runner workspace;
- public Host, CLI, Studio, and A2A contracts stay engine-agnostic.

## Concrete changes

- Added an adapter-local `entangle-opencode-session-map.json` file under the
  node engine-state root.
- Before each OpenCode turn, the adapter reads the map for the current Entangle
  `sessionId` and adds `--session <mapped-id>` when present.
- After OpenCode emits a new `sessionID`, the adapter updates the mapping for
  that Entangle session.
- Kept `engineSessionId` as the generic public result field; no OpenCode-specific
  field leaks into shared contracts.

## Tests required

- OpenCode adapter test proving the first turn writes the mapping.
- OpenCode adapter test proving a later turn passes `--session` and updates the
  stored mapping.
- Runner typecheck and lint.

## Migration and compatibility

This is additive. Existing nodes without a session map behave as before. The
map is adapter-owned runtime state and can be safely regenerated by later turns
if removed.

## Risks and mitigations

- Risk: a stale OpenCode session id could fail inside OpenCode.
  Mitigation: the failure remains bounded to the engine turn result; deleting
  the adapter-local map returns the node to fresh-session behavior.
- Risk: Entangle could become coupled to OpenCode session internals.
  Mitigation: only the adapter reads/writes the map; shared contracts continue
  to use generic `engineSessionId`.
- Risk: concurrent turns for one node could race on the map.
  Mitigation: current runner turn execution is serialized for a node; a future
  concurrent engine loop should move this map behind the runner state store.

## Audit notes

This slice adds no Host-readable runner path and no graph protocol change. The
new file is runner-local engine state, analogous to the existing isolated
OpenCode database/config roots.

## Open questions

- A later attached-server adapter should use OpenCode's session HTTP API
  directly for permission bridging, abort/resume, and live event streaming.
- Entangle may eventually expose a generic engine-session continuity policy per
  node, but the default should stay on because it improves coding-agent context.
