# Distributed Proof Profile Contract Slice

## Current Repo Truth

`pnpm ops:distributed-proof-kit` generates
`operator/proof-profile.json`, and `pnpm ops:distributed-proof-verify` can
consume that file. Before this slice, the profile shape was validated only by
ad hoc field readers inside the verifier. Malformed profiles could silently
drop bad values and fall back to defaults, which is risky for a proof that is
supposed to demonstrate a specific distributed topology.

## Target Model

The distributed proof profile should be an explicit contract. Generated proof
kits should validate the manifest they write, and the verifier should reject
malformed or internally inconsistent manifests before reading Host state. The
contract should stay compatible with the existing generated profile fields and
keep OpenCode as the default agent engine.

## Concrete Changes

- Added `distributedProofProfileSchema` and assignment-profile contracts to
  `@entangle/types`.
- Added a pure Node script-side `normalizeDistributedProofProfile` helper so
  ops scripts can validate manifests without requiring a TypeScript build step.
- The proof kit now validates the generated manifest before printing or writing
  it.
- The verifier now validates `--profile` input before resolving CLI/profile
  defaults.
- The smoke now checks the profile contract helper syntax and proves invalid
  schema versions plus inconsistent assignment runtime kinds fail before Host
  inspection.

## Migration And Compatibility Notes

Existing proof kits generated by the previous manifest slice already write
`schemaVersion: 1` and the same top-level ids, so they remain compatible. Hand
written profiles must now include `schemaVersion: 1` and use supported runtime
and agent-engine kind values instead of relying on silent fallback.

The verifier still lets explicit CLI flags override profile fields. The profile
contract validates the manifest itself; operator overrides remain the mechanism
for intentionally changing Host URLs, runner ids, node ids, or expected engine
kinds at invocation time.

## Risks And Mitigations

- Risk: strict profile validation rejects a hand-written profile that
  previously worked by accident.
  Mitigation: generated profiles remain compatible, and the failure happens
  before Host or runner state is inspected.
- Risk: the script-side helper duplicates the package-level schema.
  Mitigation: the helper is intentionally small and dependency-free so Node ops
  scripts can validate profiles without a build step; the TypeScript schema
  remains the canonical package contract.
