# Distributed Proof Verifier Slice

## Current Repo Truth

`pnpm ops:distributed-proof-kit` can prepare copyable runner directories and
operator commands for a three-runner proof across separate machines. Before
this slice, operators still had to inspect Host state manually after starting
those machines to confirm that runner registration, trust, assignment,
projection, User Client URLs, and optional conversation evidence converged.

## Target Model

The operator machine should have a single verifier command that checks a
running distributed proof through network APIs only:

- Host status exposes an active Host Authority;
- expected runner registrations exist and are trusted;
- runners are online by default, with an explicit stale-tolerant option;
- runner heartbeats include expected assignment ids;
- expected assignments are accepted or active;
- Host projection contains matching assignment and runtime records;
- User Node Human Interface Runtime projections expose User Client URLs;
- optional User Client `/health` checks pass;
- optional primary User Node to agent conversation projection exists.

The verifier must not read Host, runner, or shared workstation files.

## Concrete Changes Required

- Add `pnpm ops:distributed-proof-verify`.
- Poll Host `/v1/host/status`, `/v1/runners`, `/v1/assignments`, and
  `/v1/projection` until checks pass or timeout.
- Support the same default runner/node ids generated by the proof kit, with
  override flags.
- Support optional User Client health checks and conversation projection
  checks.
- Add JSON output for automation.
- Add an embedded self-test fixture so verifier logic can run without external
  machines or model credentials.

## Migration And Compatibility Notes

This is additive operator tooling. It reads existing Host APIs and User Client
health endpoints. Existing smoke, kit, CLI, Studio, Host, and runner behavior is
unchanged.

## Risks And Mitigations

- Risk: a verifier hides real distributed issues by reading local files.
  Mitigation: it only uses Host HTTP APIs and optional User Client HTTP health
  endpoints.
- Risk: the verifier races runner startup.
  Mitigation: it polls until timeout and exposes `--timeout-ms` plus
  `--poll-interval-ms`.
- Risk: operators need different ids.
  Mitigation: runner ids and graph node ids are all configurable and default to
  the proof kit values.

## Open Questions
